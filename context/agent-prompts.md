# Agent Prompts — Student Directory

Tech stack: Next.js 14+ (App Router) + Tailwind CSS + Supabase (PostgreSQL)

---

## Phase 1 — Foundation (all parallel, no dependencies)

| Agent | Prompt |
|-------|--------|
| **A — Data Layer** | You are building the data layer for a student housing directory in Thessaloniki, Greece. **Tech:** Supabase (PostgreSQL). **Task:** (1) Create the Supabase schema implementing a star schema. Fact table: `listings` with `listing_id` as composite key (format `LLLLNNN (7 digits, no hyphen)` — 4-digit landlord ID, 2-digit listing suffix). Dimension tables: `landlords` (name, contact info, landlord_id), `rent` (monthly_price, currency, bills_included boolean, deposit), `location` (address, neighborhood, lat, lng), `property_types` (studio, 1-bed, 2-bed, room, etc.), `amenities` (AC, furnished, balcony, elevator, parking, ground_floor, etc. — stored as a join table `listing_amenities`), `faculty_distances` (precomputed walk_minutes and transit_minutes from each listing to each faculty reference point). (2) Create a faculty reference table with all major Thessaloniki university faculties (AUTH, UoM, IHU) with coordinates. Group nearby faculties under a single reference point where appropriate. (3) Generate 10 realistic seed listings with all dimensions filled — no nulls. Output: SQL migration files in `supabase/migrations/`, a seed file in `supabase/seed.sql`, and a `schema.md` documenting every table and relationship. Work only in `supabase/` and `docs/`. |
| **B — Backend/API** | You are building the API layer for a student housing directory in Thessaloniki. **Tech:** Next.js 14+ App Router API routes + Supabase client. **Task:** (1) Initialize the Next.js project: `npx create-next-app@latest student-directory --app --tailwind --src-dir`. Configure `next.config.js` for the project. Install `@supabase/supabase-js`. Set up Supabase client in `src/lib/supabase.js` (reads `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` from env). (2) Create API route handlers in `src/app/api/`: `GET /api/listings` — accepts query params: `faculty` (string), `max_budget` (number), `types` (comma-separated), `duration` (string), `exclude_amenities` (comma-separated dealbreaker IDs). Returns filtered listings with the relevant faculty distance columns. Supports `sort_by` and `sort_order` params. `GET /api/listings/[id]` — returns full listing detail with all dimensions, all amenity names, all faculty distances, landlord info. `GET /api/faculties` — returns all faculty reference points (for quiz dropdown). (3) For now, stub these with hardcoded mock JSON responses matching the schema from Agent A. The mock data shape must match the Supabase schema exactly so swapping to real queries later is trivial. Output the response shape contracts in `docs/api-contracts.md`. Work only in `src/app/api/`, `src/lib/`, and `docs/`. |
| **C — Frontend/UI** | You are building the frontend for a student housing directory in Thessaloniki. **Tech:** Next.js 14+ App Router + Tailwind CSS. **Design:** Minimalist, Bauhaus-inspired. Sans-serif fonts (Inter body, Montserrat headings via Google Fonts). Navy `#1B2A4A` primary, Gold `#C4953A` accent, White `#FFFFFF` background, Dark gray `#1F2937` text. Mobile-first, generous whitespace. Soft CTAs ("Find housing", "View details"). **Task:** (1) Build the layout shell in `src/app/layout.js`: import fonts, set metadata (title: "Thessaloniki Student Housing Directory"), global styles. (2) Build `src/components/Navbar.js`: sticky top, site name left, nav links right, hamburger + slide-out drawer on mobile. (3) Build `src/components/Footer.js`: dark background, site info, copyright with dynamic year. (4) Build the quiz page at `src/app/page.js`: a multi-step or single-page form with these fields — Faculty (dropdown, will be populated from API later, hardcode 5 options for now), Rental duration (select), Budget upper bound (number input with slider), Property type (multi-select checkboxes: Studio, 1-Bed, 2-Bed, Room), Dealbreakers to avoid (multi-select checkboxes: Ground floor, No AC, Bills not included, No elevator, etc.). On submit, store selections in state/URL params and navigate to `/results`. (5) Create placeholder pages at `src/app/results/page.js` and `src/app/listing/[id]/page.js` with "Coming in Phase 2" text. Work only in `src/app/` (except `api/`), `src/components/`, and `public/`. |

## Phase 2 — Core Logic (A is independent; B needs A's schema; C builds against B's mocks)

| Agent | Prompt |
|-------|--------|
| **A — Data Layer** | Continue building the data layer. **Task:** (1) Build a Python ingestion script (`scripts/ingest.py`) that takes a CSV or JSON batch file and: validates all required fields are present, assigns `listing_id` (format `LLLLNNN (7 digits, no hyphen)`) if not already set, normalizes data (standardize price to EUR, clean address strings, validate coordinates), flags any listing with missing dimensions in a `_validation_errors.json` output, upserts valid records into the Supabase tables. (2) Build a distance precomputation script (`scripts/compute_distances.py`) that: reads all listings and all faculty reference points from Supabase, calls a routing API (Google Maps or OSRM — prefer OSRM for free usage) to compute walk_minutes and transit_minutes for each listing→faculty pair, writes results to the `faculty_distances` table. (3) Create a batch template file (`templates/batch_template.csv`) showing the expected input format with column headers and 2 example rows. Work only in `scripts/`, `templates/`, and `supabase/`. |
| **B — Backend/API** | Continue building the API layer. The Supabase schema is now finalized (read `supabase/migrations/` and `docs/schema.md` for the exact table structure). **Task:** (1) Replace all mock stubs with real Supabase queries. `GET /api/listings`: query the `listings` fact table joined with all dimensions. Filter by `max_budget` (rent.monthly_price <=), `types` (property_type IN), `exclude_amenities` (listings that do NOT have these amenities — inverse filter via NOT EXISTS on `listing_amenities`), `duration`. Join `faculty_distances` filtered to the selected `faculty` param. Support `sort_by` (price, distance_walk, distance_transit) and `sort_order` (asc, desc). `GET /api/listings/[id]`: full join across all dimension tables + all faculty distances + all amenities. `GET /api/faculties`: select all from faculty reference table. (2) Add error handling: 400 for invalid params, 404 for missing listing, 500 with generic message. (3) Update `docs/api-contracts.md` with final response shapes. Work only in `src/app/api/`, `src/lib/`, and `docs/`. |
| **C — Frontend/UI** | Continue building the frontend. API contracts are defined in `docs/api-contracts.md`. **Task:** (1) Build `src/app/results/page.js`: read quiz selections from URL search params, call `GET /api/listings` with those params, display results in a sortable table/card list showing: listing photo thumbnail, address/neighborhood, monthly price, property type, "Time to [faculty]" (walk + transit), status pills for key amenities. Each column header is clickable to sort asc/desc. Each row/card links to `/listing/[id]`. Show a "No results" empty state if nothing matches. (2) Build `src/app/listing/[id]/page.js`: call `GET /api/listings/[id]`, display: photo gallery (grid or carousel), price prominently, all dimensions in a clean info grid, amenities as pill/badge list, all faculty distances in a table, description text, "Contact landlord" button (disabled, labeled "Coming soon"). (3) Build `src/components/ListingCard.js` as a reusable card for the results grid. Work only in `src/app/` (except `api/`), `src/components/`, and `public/`. |
